01 The Microsoft AI Footprint — and where Foundry sits

Microsoft's AI is not one product; it's a stack. The SaaS Copilots you can't reshape sit on top, the managed model APIs in the middle, and the raw silicon and safety systems at the bottom. Azure AI Foundry is the layer where Microsoft hands you the controls — the same models powering M365 Copilot become building blocks you compose, ground, secure, and ship.

02 What Azure AI Foundry actually is

Foundry (GA-rebranded from Azure AI Studio at Ignite 2024) is a unified application platform — a portal (ai.azure.com), an SDK, and a set of first-class Azure resources for building, evaluating, deploying, and operating generative-AI apps and agents.

Two ways to provision — know which one you have

Hub-based projects (AML lineage)

The original model: an AI Foundry hub (a kind of Azure Machine Learning workspace) that provisions a set of dependency resources and parents one or more projects. The hub centralizes networking, connections, compute, and security so projects inherit them.

  • Hub auto-creates: Storage, Key Vault, Container Registry (optional), Application Insights (optional), and an Azure AI Services/AOAI resource.
  • Best when teams share network + connections and you want AML features (managed compute, prompt flow runtimes).

Foundry projects (new resource)

The newer Azure AI Foundry resource (Microsoft.CognitiveServices/accounts, kind AIServices) hosts Foundry projects as a first-class, lighter-weight object — the default for the Agent Service and the unified Foundry API.

  • Fewer mandatory dependencies; agent threads/state can be Microsoft-managed or BYO (your Cosmos DB, Storage, AI Search).
  • Best for agent-centric and "just give me an endpoint" scenarios.
Ops gotcha: the two project types differ in RBAC surface, networking story, and which SDK/API endpoint you call. Standardize on one per platform team and encode the choice in your landing-zone bicep/Terraform module so app teams can't drift.

The model catalog — one estate, many providers

A single governed catalog of 1,900+ models spanning Azure OpenAI (GPT-4o/4.1, o-series, GPT-image), Microsoft Phi, Meta Llama, Mistral, Cohere, Nvidia NIM, Stability, and the curated Hugging Face collection.

How a model becomes an endpoint

Serverless API (MaaS)

Pay-per-token REST endpoint. No infra to manage, fastest to ship, but inference runs in a Microsoft-managed pool. Lock down with key rotation + private endpoint where supported.

Managed Compute

Model runs on dedicated GPU VMs in your deployment. Full VNet isolation, autoscale, you own patching cadence of the inference container image. Highest control + cost.

Azure OpenAI deployment

Provisioned-Throughput Units (PTU) for predictable latency/SLA, or standard pay-go. The workhorse for Copilot-grade workloads with data-residency commitments.

The Foundry build lifecycle

  1. Explore — model catalog, benchmarks, model leaderboard, try in playground.
  2. Ground — connect data (Azure AI Search vector index, your storage, Fabric, Bing) → RAG.
  3. Build — prompt flow / Agent Service / SDK; add tools (function calling, code interpreter, OpenAPI, Logic Apps).
  4. Evaluate — built-in evaluators (groundedness, relevance, fluency, safety) + your own datasets; AI red-teaming agent (PyRIT-based).
  5. Deploy — choose deployment mode, wire content safety, set rate limits.
  6. Observe — tracing, Application Insights, online evaluation, drift & abuse monitoring.
Secure-SDLC mapping: Evaluate + red-team = your pre-prod security gate; Observe = detection & response. Treat eval thresholds as release-blocking quality gates in CI/CD.

04 Identity & Entra ID integration

Foundry runs on the Cognitive Services / AML resource providers, so it inherits the full Entra ID + Azure RBAC machinery. The design goal: zero standing secrets — managed identities and RBAC data-plane roles instead of API keys.

The auth chain

  • User → portal/app: Entra interactive sign-in; enforce Conditional Access (device compliance, MFA, sign-in risk) on the app registration.
  • App → Foundry endpoint: prefer Entra token (DefaultAzureCredential) over keys. Disable local/key auth where policy allows (disableLocalAuth).
  • Foundry → dependencies: the workspace/account system- or user-assigned managed identity authenticates to Storage, Key Vault, AI Search, Cosmos — granted via RBAC, never connection strings.
  • Agent → tool/API: On-Behalf-Of (OBO) where the agent must act as the user; otherwise the agent's own identity with least-privilege scope.
Prompt-injection caveat: if an agent holds a powerful identity and ingests untrusted content, prompt injection becomes privilege escalation. Scope the agent's identity to exactly the tools it needs and gate high-impact actions behind human approval or a separate trust boundary.

Separate control-plane (manage the resource) from data-plane (call the model) grants. App identities almost never need control-plane roles.

Least-privilege patterns that survive an audit

PIM for control plane

Owner/Contributor on Foundry resources is eligible-only via Privileged Identity Management, activated with justification + approval.

Per-project boundaries

One project = one trust boundary. Map projects to app teams; RBAC at the project scope, not the subscription.

No keys in code

Workload identity federation for CI/CD; managed identity at runtime. Key Vault references for any unavoidable secret, with rotation.

05 Network architecture — and both meanings of "WAF"

In a landing-zone conversation "WAF" is overloaded. This section disambiguates: the Web Application Firewall that protects your exposed inference endpoints (here), and the Well-Architected Framework design pillars (§7). Both matter; they are not the same thing.

Network isolation model

  • Managed VNet: the Foundry hub provisions a Microsoft-managed VNet for its compute. Three modes: public, private w/ internet outbound, private w/ approved outbound only (FQDN allowlist).
  • Private Endpoints: ingress to the hub/project, AOAI, AI Search, Storage, Key Vault, Cosmos — all via privatelink PEs in the spoke. Public network access set to Disabled.
  • Private DNS: link the privatelink.openai.azure.com, privatelink.cognitiveservices.azure.com, privatelink.api.azureml.ms, privatelink.search.windows.net, etc. zones to the hub (centralized DNS).
  • Egress control: Azure Firewall (or the managed VNet's approved-outbound list) constrains what the workload can reach — critical because models/agents can be coerced into outbound calls.

The WAF (Web Application Firewall) tier

When you expose a chat/agent endpoint to users or partner apps, never publish the model endpoint directly. Front it:

  • Azure Front Door + WAF (global, anycast, TLS offload, OWASP CRS + bot manager) for internet-facing apps, or App Gateway + WAF (regional, in-VNet) for internal/regulated.
  • Azure API Management as a GenAI Gateway behind the WAF: token-based rate limiting (TPM/RPM), load-balancing & failover across multiple AOAI/PTU deployments, semantic caching, request/response logging, and emitting token-usage metrics for chargeback.
  • Defense-in-depth chain: Front Door+WAF → APIM → Private Endpoint → Foundry → Content Safety → model.
WAF rules ≠ prompt firewall. An OWASP WAF stops SQLi/XSS at the HTTP layer; it does not understand prompt injection. Pair it with Azure AI Content Safety Prompt Shields and groundedness checks for the LLM-specific threats.

06 Landing into the Azure Landing Zone

A Foundry workload is an application landing zone — a subscription (or pair) vended from your management-group hierarchy, peered to the platform's connectivity hub, governed by inherited policy. Here's the mapping.

Management group & subscription placement ▶

Place AI app subscriptions under the Landing Zones → Corp (or Online if internet-facing) management group so they inherit the corp policy baseline. Use subscription vending to provision them with networking, identity, and policy pre-wired. Keep platform resources (firewall, DNS, ExpressRoute/VPN) in the Platform → Connectivity subscription — Foundry spokes consume, never host, them.

Connectivity topology ▶

Hub-spoke or Virtual WAN. The Foundry spoke holds private endpoints; the hub holds Azure Firewall, the WAF tier, and the centralized Private DNS zones. Crucial detail: AI private-link FQDNs must resolve through the central DNS — a missing or unlinked privatelink.openai.azure.com zone is the single most common "it works in portal, fails from the app" outage.

Governance via Azure Policy ▶
  • Deny public network access on Cognitive Services / AML / Search / Storage.
  • Deny or Audit account creation outside approved regions (data-residency / sovereignty).
  • DeployIfNotExists diagnostic settings → central Log Analytics.
  • Audit customer-managed keys (CMK) on all stateful dependencies.
  • Restrict allowed models / deployment SKUs to a vetted list to control cost and supply-chain risk.
Reference accelerators to start from ▶

Don't hand-roll. Start from Microsoft's "Azure OpenAI / AI Foundry chat baseline" reference architecture, the GenAI Gateway APIM reference, and the ALZ Bicep / Terraform accelerators. Wrap Foundry provisioning in an IaC module your platform team owns, parameterized for network mode, CMK, and RBAC so app teams get a compliant project by default.

08 Foundry Agent Service

The managed runtime for stateful, tool-using agents — the production path beyond a raw chat-completions call. It owns threads, runs, tool orchestration, and (optionally) your bring-your-own state and network.

What it gives you

  • Threads & runs — managed conversation state and execution loop (OpenAI Assistants-compatible surface).
  • Tools — file search, code interpreter, function calling, OpenAPI 3.0 tools, Azure AI Search, Bing grounding, Logic Apps / Azure Functions actions.
  • Connected / multi-agent — agents that call other agents; orchestration patterns for decomposition.
  • BYO resources — your Cosmos DB (thread state), Storage (files), and AI Search (knowledge) inside your VNet, rather than Microsoft-managed defaults — the choice for regulated data.

Security posture for agents

  • BYO VNet / private network injection so agent egress is your egress (Firewall-filtered).
  • Least-privilege tool identity — each tool/connection gets only the RBAC it needs; OBO for user-context actions.
  • Content Safety + Prompt Shields on inputs and tool outputs; treat tool-returned content as untrusted.
  • Tracing every run to App Insights — agent decisions and tool calls are your audit trail and incident-response timeline.
Threat-model the toolset, not just the model. The blast radius of an agent equals the union of its tools' permissions. A read-only data tool is low risk; a Logic App that can send mail or hit an internal API is a lateral-movement primitive if the agent is hijacked.

09 Data governance, safety & observability

The controls that turn a working prototype into something legal, security, and your auditors will sign off on.

Data handling

  • AOAI: prompts/responses not used to train base models; your data stays in your resource.
  • CMK on Storage / KV / Cosmos / Search for encryption-at-rest you control.
  • Data residency via region pinning + policy; check per-model MaaS terms separately.
  • Purview for cataloging & DLP over grounding sources.

Responsible AI & safety

  • Azure AI Content Safety: hate/violence/sexual/self-harm filters, severity tuning.
  • Prompt Shields (jailbreak + indirect injection), protected-material detection, groundedness detection.
  • Built-in + custom evaluations and the AI red-teaming agent as release gates.

Observability & defense

  • Tracing + Application Insights for token usage, latency, and run inspection.
  • Microsoft Defender for Cloud — AI workload threat protection / posture.
  • Diagnostic logs → central Log Analytics → Sentinel analytics for abuse & anomaly detection.
  • Online evaluation to catch drift and groundedness regressions in prod.
The one-paragraph takeaway. Azure AI Foundry is where Microsoft's AI stops being a feature you consume and becomes a platform you own. That ownership is the whole point — and the whole risk. Land it as a governed application landing zone, authenticate everything with managed identity + Entra RBAC, isolate it with private endpoints behind a real WAF, gate releases on evaluations and red-teaming, and observe it like any other production system. Do that, and the same models behind M365 Copilot become a build surface that satisfies all five Well-Architected pillars.
