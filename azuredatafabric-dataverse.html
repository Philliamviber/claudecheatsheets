01 The Microsoft data estate — and where these two sit

Microsoft's data story converged on two SaaS pillars: Microsoft Fabric (analytics, with OneLake as the single logical lake) and Dataverse (the transactional, low-code data backbone under Power Platform & Dynamics 365). Both are tenant-scoped, Entra-governed, and increasingly stitched together by open Delta-Parquet and Purview.

All bill against a single Fabric capacity (F SKU) — a shared, burstable compute pool. Capacity is a governance & cost boundary, not a security boundary.

OneLake — "OneDrive for data"

  • One per tenant, hierarchical: tenant → workspaces → items (lakehouses, warehouses) → files/tables in Delta-Parquet.
  • Shortcuts: reference data in ADLS Gen2, S3, GCS, Dataverse, or another workspace without copying — powerful and a governance trap (data crosses boundaries by reference).
  • Mirroring: near-real-time replicas of Azure SQL, Cosmos DB, Snowflake into OneLake.
  • OneLake security (OneSecurity): define table/row/column security once at the lake, enforced across engines.
  • Open format means external tools (and external risk) can read it — classification + access control matter more, not less.

Dataverse — the structured backbone

Containers

  • Environments — the top isolation boundary (dev/test/prod, geo, with/without a Dataverse DB).
  • Solutions — ALM packaging of tables, apps, flows.
  • Tables / columns / relationships — the relational model; standard + custom.

Access primitives

  • Business units + security roles (privilege × access level).
  • Teams (owner / access / Entra-group / managed).
  • Field-level security, hierarchy security, record sharing.

Dataverse is OLTP and permission-rich — its security model (§5) is far more granular than Fabric's workspace roles, and it's where most real-world over-permissioning hides.

Stitching the two together

  • Link to Microsoft Fabric — surfaces Dataverse tables as OneLake shortcuts: zero-ETL, near-real-time, analytics-ready. The modern path (replaces Export to Data Lake).
  • Azure Synapse Link for Dataverse — continuous export of Dataverse to ADLS Gen2 / Synapse for teams not yet on Fabric.
  • Virtual tables — Dataverse reads external sources live without import.
  • Dual-write — bi-directional sync between Dataverse and Dynamics 365 F&O.
Hardening note: Link to Fabric inherits Dataverse schema but not its row/field security automatically — once data lands in OneLake, re-apply classification and OneLake/workspace access control or you've quietly widened the audience.

04 Identity, workspace roles & service principals

Both pillars authenticate through Entra ID. Fabric authorizes coarsely at the workspace, then finely at the data; Dataverse authorizes through a deep role model (§5). The first hardening win is replacing per-user grants with Entra-group-backed roles.

Fabric workspace roles

RoleGrants
AdminFull control incl. access mgmt + delete workspace
MemberAdd others (≤ Member), share items, edit
ContributorCreate/edit items; no access mgmt
ViewerRead built reports / query with RLS applied

Workspace roles grant broad data access. For least privilege, keep most users at Viewer and enforce row/column security at the lake/warehouse, not the workspace.

Non-human identity

  • Workspace identity — a managed identity for the Fabric workspace; use it for trusted access to ADLS/Key Vault instead of stored credentials.
  • Service principals — for CI/CD and automation; gate behind a tenant setting that limits which security groups may use SPN access.
  • Dataverse application users — register the SPN as an application user with a least-privilege security role; never reuse the System Administrator role for integrations.
  • Prefer workload identity federation over client secrets for pipelines; if a secret is unavoidable, store it in Key Vault with rotation.
Conditional Access is the front door for both. Power BI/Fabric and Power Platform are CA-targetable cloud apps. Require MFA + compliant device, block legacy auth, and consider sign-in-risk policies. CA is the single highest-leverage control over this entire estate.

05 The Dataverse security model — layer by layer

Dataverse access is the intersection of several independent layers. A user can do something only if every layer permits it. Understanding the order is the difference between least privilege and an accidental tenant-wide read.

1 · Environment & business units ▶

The environment is the outermost boundary — separate dev/test/prod and, where law requires, by geography. Within it, business units form a hierarchy that scopes data ownership. Access levels (User → BU → parent-child BU → Organization) are defined relative to this tree, so BU design directly shapes how wide "Business Unit"-level access reaches.

2 · Security roles (privilege × access level) ▶

A security role is a matrix: for each table and each action (Create/Read/Write/Delete/Append/AppendTo/Assign/Share) it sets an access level. Roles are additive — a user's effective rights are the union of all assigned roles, so a single over-broad role silently elevates everyone who also holds it. Audit for Organization-level Read on sensitive tables.

3 · Teams & Entra group teams ▶

Assign roles to teams, not individuals, for scale. Entra group teams bind a Dataverse team to an Entra security group so membership (and therefore access) is governed in one place — the recommended pattern for joiner/mover/leaver hygiene.

4 · Field (column) security ▶

For sensitive columns (SSNs, comp, health), enable column security and grant via a column security profile — independent of table-level rights. This is how you let users see a record but not its most sensitive fields.

5 · Hierarchy security & record sharing ▶

Hierarchy security (manager/position) grants managers visibility into reports' records. Record sharing grants ad-hoc access to specific rows — convenient but invisible to role audits, so monitor and prefer team ownership over sprawling shares.

Effective access = environment ∩ security roles ∩ field security ∩ (sharing/hierarchy). Review it from the data outward ("who can read table X?"), not the user inward — the additive role model makes user-by-user review miss aggregate exposure.

06 Network isolation — for a SaaS you don't host

You can't put a SaaS in your VNet, but you can control how traffic reaches and leaves it. Fabric and Dataverse expose distinct private-networking controls — know which applies where.

Microsoft Fabric

  • Private Link (inbound) — tenant/workspace-level private endpoint so users reach Fabric only over your network; pair with the tenant setting to block public internet access.
  • Managed Private Endpoints (outbound) — Fabric reaches your private data sources (Azure SQL, Storage) without public exposure.
  • Trusted workspace access + workspace identity — let firewalled ADLS Gen2 trust a specific Fabric workspace.
  • Managed VNet isolates Spark compute. Note: enabling Private Link has feature trade-offs — validate workloads.

Dataverse / Power Platform

  • IP firewall — restrict Dataverse access to known IP ranges; IP cookie binding to resist token replay from other IPs.
  • VNet support (subnet delegation) — route Power Platform outbound (plug-ins, connectors) through your VNet for egress control.
  • Tenant isolation — block inbound/outbound cross-tenant connector traffic to stop data exfil to foreign tenants.
  • Customer Lockbox — require explicit approval before Microsoft engineers can access your data during support.
Shortcuts & connectors cross boundaries by design. A OneLake shortcut or a Power Platform connector can pull data across the network perimeter you just built. Network controls are necessary but not sufficient — pair them with DLP (connectors) and shortcut governance (OneLake).

09 Monitoring, audit & incident response

SaaS means you can't read host logs — your visibility comes from the platform's own audit streams. Wire them to a SIEM and treat them as your only timeline.

Audit sources

  • Purview / M365 audit log — Fabric & Power BI activity events (view, export, share, permission change).
  • Dataverse auditing — record + field change history and access logs; enable per-table.
  • Power Platform activity logging & admin analytics.
  • Capacity Metrics app for Fabric throttling/health.

Detection

  • Microsoft Defender for Cloud Apps — anomaly + session policies over Power BI/Fabric.
  • Forward audit logs to Microsoft Sentinel; alert on mass export, sharing to externals, role escalation, DLP override.
  • Watch for new shortcuts to external storage and new tenant-isolation exceptions.

Respond

  • Pre-stage runbooks: revoke sharing, disable SPN, quarantine environment, rotate keys/CMK.
  • Customer Lockbox approvals are themselves an audit event — review them.
  • Use environment/workspace as a containment boundary; have an export-blocking break-glass DLP policy ready.
The one-paragraph takeaway. Fabric and Dataverse hand you a unified, open, tenant-wide data estate — and with SaaS, almost your entire security job collapses onto identity, authorization, classification, exposure, and monitoring. Back roles with Entra groups and Conditional Access, harden the tenant with Managed Environments + DLP + tenant isolation, classify and encrypt the data (labels, CMK, RLS/CLS/field security), pull the perimeter in with Private Link and managed endpoints, and stream every audit log to Sentinel. Do that and one lake serving every workload becomes an asset you can defend — not a single pane of glass into everything you failed to lock down.
